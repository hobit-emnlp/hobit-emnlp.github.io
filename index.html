hoBIT acquires a compact session profile by dual-asking, then uses it as both a soft query prefix and a hard metadata filter.

Why hoBIT?

Academic advising is profile-dependent retrieval: semantic similarity alone is not enough.

Profile-blind RAG misses context

Graduation, course, and major rules vary by department, admission cohort, and major type, so the same query can require different documents.

Profile metadata is indexed offline

Chunks are tagged with department, admission cohort, and required profile fields before retrieval.

Dual-asking keeps interaction light

The bot asks for missing fields only when they are needed, then reuses them inside the session.

Same Question, Different Correct Answer

Like realistic search snippets, curriculum documents can look near-identical. hoBIT exposes the hidden profile signal that decides which evidence is correct.

Profile-Blind RAG

"Can you tell me my graduation requirements?"

Semantically similar document

Wrong cohort evidence, wrong answer, or insufficient user information.

hoBIT Profile-Aware RAG

Dept: CS, Cohort: 2020; graduation requirements?

Profile-matched source evidence

Correct cohort evidence, source-tracked answer, reusable session profile.

Methodology

hoBIT combines intent routing and profile-aware RAG, with profile signals introduced before indexing and again at retrieval time.

1

Intent Classification and RAG Routing

hoBIT first decides whether a message should be handled directly or routed into retrieval. Greeting, FAQ, ability, and small-talk intents avoid unnecessary generation; retrieval intents enter the profile-aware RAG pipeline.

greetingFAQabilitysmall talkretrieval
Only retrieval queries need source-grounded evidence and profile-conditioned search.
2

Profile-Based Preprocessing

Static and dynamic school documents are chunked, tagged with profile metadata, and indexed before deployment. The tags let retrieval distinguish near-identical curriculum documents from different departments, admission cohorts, and major types.

PDFschunksprofile tagsQdrant
Tags include department, admission_year, and required_profile fields used by reactive dual-asking.
3

Profile Injection at Retrieval Time

At runtime, hoBIT acquires missing profile fields through dual-asking and injects the session profile in two places: as a soft prefix that steers ranking, and as a hard payload filter that removes wrong-profile evidence.

Dept: CSCohort: 2020Major type: Intensive
The soft prefix improves recall, while the hard filter controls eligibility.

hoBIT System Overview

Offline profile-based data preprocessing meets online profile-based RAG with dual-asking, soft prefix injection, and hard payload filtering.

Runtime Pipeline: Four Stages

Intent classification, filter extraction, profile validation, and source-tracked answer generation run around a session profile.

Results

Profile injection dominates the retrieval stack and makes profile-blind boosters redundant.

Source Match 0.070 -> 0.585

Adding profile injection produces the largest jump in the deployed ablation.

Overall PASS 6.8% -> 47.4%

Dual-asking plus profile-aware retrieval makes correct answers reachable.

Intent routing 96.0% acc.

The retrieval-intent gate reaches 0.990 F1, so only retrieval queries enter the profile-aware pipeline.

Open-ended advising 0.796 overall

The ground-truth-free track reaches 59.1% Overall PASS with the deployed pipeline.
