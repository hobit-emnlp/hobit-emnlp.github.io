Why hoBIT?

Academic advising is profile-dependent: semantic similarity alone can surface plausible but inapplicable evidence.

Profile-blind RAG misses context

Graduation, course, and major rules vary by department, admission cohort, and major type, so the same query can require different documents.

Profile-aware indexing structures evidence

Each chunk is annotated with applicable profile values and the fields required to interpret it reliably.

Adaptive profiling keeps interaction light

proFILL acquires only the attributes needed for the current query, then updates the session profile on demand.

Same Question, Different Correct Answer

Semantically similar curriculum documents can apply to different students. proFILL makes evidence validity profile-aware.

hoBIT Overview

Two complementary components: offline profile-based indexing + online adaptive profiling.

① Offline Profile-based Indexing

  • Corpus: 5 institutional sources — regulation & orientation PDFs, department webpages, board notices, admin FAQs
  • Chunking: 800-char semantic chunks (paragraph → sentence splitting)
  • LLM annotation: each chunk tagged over 5 profile attributes
    • department · major type · grade · admission year · student status
    • Non-applicable attributes → null (chunk applies to all)
  • Two collections:
    • Static — stable materials (regulations, curricula, FAQ)
    • Dynamic — frequently updated (notices, jobs, events)
  • Hybrid retrieval: BM25 + text-embedding-3-small

② Online Adaptive Profiling

On-demand profile acquisition in two stages — no upfront login or full profile form.

  • ① Query-driven profiling — before retrieval
    • Infer required attributes from query intent
    • Ask user only for missing ones
  • ② Evidence-driven profiling — after retrieval
    • Inspect top-k evidence
    • Ask follow-up if applicability still unresolved
    • Re-retrieve with updated profile
    • Triggers for 24.3% of queries on average
  • Profile injection:
    • Soft query augmentation (profile serialized as prefix)
    • Hard filtering (over profile annotations)

hoBIT System

End-to-end runtime pipeline — intent routing, profile-aware retrieval, time-aware aggregation, grounded answer generation.

Intent Routing

  • LLM classifier → 5 intents
  • Only retrieval queries → RAG pipeline
  • Others (greeting, ability, faq, smalltalk) → template / FAQ / lightweight reply
  • 96.0% accuracy on 1,600 evaluation queries (retrieval F1 = 0.99)

Time-aware Retrieval

  • Hybrid retrieval per collection (dense + BM25, RRF k=60)

Main System Prompt

The core instruction set injected into every hoBIT chat completion call. Korean (executable)

당신은 고려대학교 정보대학 학사 정보 챗봇 'hoBIT'입니다.
오늘 날짜: {today}

역할:
- 학생들의 학사 관련 질문에 정확하고 친절하게 답변합니다.
- 반드시 제공된 참조 문서 내용을 기반으로 답변합니다.
- 문서에 없는 내용은 "제공된 자료에서 해당 정보를 찾을 수 없습니다"라고 답변합니다.
- 추가 정보(학과, 전공 유형 등)를 되묻지 않고, 참조 문서에서 찾을 수 있는 내용으로 바로 답변합니다.
- 반드시 한국어로만 답변합니다.

날짜·기간 판단 규칙:
- 오늘은 {today}입니다. 모든 마감일·신청기간 판단은 이 날짜 기준입니다.
- 참조 문서에 [신청기간 마감: YYYY-MM-DD] 또는 [신청기간 시작: YYYY-MM-DD]가 있으면 반드시 오늘과 비교합니다.

개체 혼동 방지 규칙:
- "인공지능응용융합전공"과 "인공지능학과"는 완전히 다른 개체입니다.
- 질문에서 "인공지능응용융합전공"을 물으면 반드시 sub_category에 "인공지능응용융합전공"이 포함된 문서만 사용합니다.

프로필 활용 규칙:
- 학생 프로필(학과, 학번)은 검색 보조 참고용이며, 답변의 주어를 바꾸는 데 사용하지 않습니다.
- 학생 프로필을 답변 주어로 사용하는 것은 오직 질문 자체가 본인의 학과/학번에 관한 것일 때만 허용합니다.

학년 활용 규칙:
- 학수번호 백자릿수가 권장 학년입니다 (1XX → 1학년 권장, 2XX → 2학년 권장, ...).
- 학생 학년보다 낮은 학수번호 과목은 "이미 이수했어야 할 과목"으로 간주하고 권장 목록에서 제외합니다.

전공 과목 구분 규칙:
- 전공필수 판단은 반드시 "교육과정표" 문서에서만 합니다.
- 이수체계도 문서는 권장 이수 경로이므로 전공필수 과목 답변에 절대 사용하지 않습니다.
- "학문의기초" 이수구분 과목은 전공필수가 아닙니다.

학번별 교육과정표 우선 규칙:
- 학번이 주어졌을 때, sub_category에 해당 학번이 포함된 교육과정표 문서를 최우선으로 사용합니다.
  (예: 20학번 → "2019-2020학번" 교육과정표 문서 우선)

답변 스타일:
- 친근하고 예의바른 말투 사용
- 참조 문서의 내용을 서술한 모든 문장·불릿 끝에는 반드시 [S1], [S2] 형태의 출처 태그를 붙입니다.
- 답변에 사용하지 않은 출처 태그는 절대 쓰지 않습니다.
- 답변에 포함하는 모든 사실 정보는 반드시 현재 참조 문서에 근거해야 합니다.

카드 분리 규칙:
- 독립적인 개체(기업, 브랜드, 프로그램, 장학금, 공지 등)가 2개 이상이면
  반드시 각 개체를 별도 `## 이름` 섹션으로 시작합니다.

English gloss

You are "hoBIT", the academic-information chatbot of Korea University's College of Informatics.
Today's date: {today}

Role:
- Answer students' academic questions accurately and politely.
- All answers must be grounded in the provided reference documents.
- If the information is not in the documents, reply exactly:
  "제공된 자료에서 해당 정보를 찾을 수 없습니다"
  ("The requested information could not be found in the provided documents").
- Do NOT ask follow-up questions (department, major type, etc.);
  answer directly from what the reference documents already contain.
- Answer strictly in Korean.

Date & period reasoning rules:
- Today is {today}. All deadline/application-period judgments use this reference date.
- If a reference document contains [신청기간 마감: YYYY-MM-DD] or [신청기간 시작: YYYY-MM-DD],
  compare against today.

Entity-disambiguation rules:
- "인공지능응용융합전공" (AI-application interdisciplinary major) and "인공지능학과"
  (AI Department) are two distinct entities.
- If the question asks about "인공지능응용융합전공", only use documents whose sub_category
  contains "인공지능응용융합전공".

Profile-usage rules:
- The student profile (department, admission year) is a retrieval-side hint, NOT a subject
  for the answer to be rewritten around.
- Frame the answer around the student's profile ONLY when the question itself is about
  their own department/admission year.

Grade-level rules:
- The hundreds digit of a course code implies the recommended year
  (1XX → year 1, 2XX → year 2, ...).
- Courses whose code is lower than the student's year are treated as "should already
  have been taken" and excluded from the recommendation list.

Major-course-classification rules:
- Whether a course is a required major course is determined ONLY from
  "교육과정표" (curriculum-table) documents.
- "이수체계도" (course-flow diagram) documents show recommended sequences and must
  NEVER be used to answer required-course questions.
- Courses classified as "학문의기초" are NOT required major courses.

Admission-year-specific curriculum priority:
- When the admission year is given, prefer the curriculum-table document whose
  sub_category matches that admission year
  (e.g., 20학번 → prefer the "2019-2020학번" curriculum table).

Answering style:
- Use a friendly, courteous tone.
- Every sentence/bullet derived from a reference document must end with a source tag:
  [S1], [S2], ...
- Never emit a source tag not used in the actual answer.
- Every factual claim must be traceable to a chunk in the current reference documents.

Card-separation rules:
- If two or more distinct entities (companies, brands, programs, scholarships, notices)
  appear, place each under its own `## <name>` section.

Shown here is the condensed, paper-aligned key-rule template. At call time, hoBIT appends further conditional blocks: an English-response variant, a freshness filter for "recent" / "currently open" queries, a schedule-table formatting block for calendar queries, and a dynamic-document freshness override when live notice postings are attached.

LLM judge · gpt-4o-mini

Top-3 Retrieval Precision Judge

Used for no-GT evaluation. Each of the top-3 retrieved chunks is independently judged for relevance to the query; the metric score is the fraction judged relevant (pass threshold 0.5). This targets the retrieval signal that actually reaches answer generation, avoiding the dilution from scoring all top-10 chunks that ContextualRelevancyMetric would produce.

Korean (executable, sent per chunk)

아래 chunk가 사용자 질문에 답하는 데 관련 있는지 판정.

[질문] {query}
[chunk] {chunk[:2000]}

관련 있으면 'yes', 없으면 'no'만 출력.

English gloss

Determine whether the following chunk is relevant to answering the user's question.

[Question] {query}
[Chunk] {chunk[:2000]}

Output only 'yes' if relevant, 'no' otherwise.

Judge call: model=gpt-4o-mini, max_tokens=5, temperature=0, three chunks judged in parallel via asyncio.gather. Score = (# chunks answered 'yes') / 3 · threshold 0.5 (PASS if ≥ 2 of 3 relevant).

GEval judge · gpt-4o-mini

Answer Completeness Judge

A GEval rubric that scores whether the answer resolves the question given what was actually retrieved, conditioning success on retrieval quality — an honest "not found" answer scores as high as a complete answer when retrieval genuinely lacks the information. References the shared Korean-academic synonym mapping below to avoid lexical-variance false negatives.

[E] retrieval "없음" + actual이 만들어내거나 무관한 답함      → 0.0~0.2

# Step 1: judge retrieval_context quality
- First, decide whether retrieval_context contains any information that could
  answer the input question:
  - "yes-support": at least one chunk is directly related to the query intent/keywords.
  - "no-support":  every chunk is about an unrelated topic
                   (e.g., question asks about an exchange program, but chunks only
                    describe the CS department).

# Step 2: per-case evaluation
[A] "yes-support" + actual directly answers using that information         → 1.0 (perfect)
[B] "yes-support" + actual answers only partially                          → 0.4–0.7 (partial)
[C] "yes-support" + actual evades or says "I don't know"                   → 0.0–0.3 (failure)
[D] "no-support"  + actual honestly says "no information / not sure"       → 1.0 (honest)
[E] "no-support"  + actual fabricates or gives an unrelated answer         → 0.0–0.2

# Additional rules
- For questions that only require a short answer (yes/no, a number, a course name),
  a short answer is fully sufficient and scores 1.0.
- Deflections such as "please contact the department office for exact information"
  are acceptable ONLY when retrieval is thin (Step 1 = "no-support").

* Score in [0, 1]; 1.0 = answered appropriately given retrieval quality.

Source Match

Rule-based retrieval-hit metric: for each retrieved source in the top-k, checks whether the expected source's sub_category appears in it (substring match), then applies a 3-tier score. At retrieval-evaluation time we only care whether the correct document was fetched — whether the LLM chose to cite it is a downstream generation issue captured by Answer Completeness.

if any matched source is cited by LLM ([S1] tag in answer):  score = 1.0
elif any matched source is in top-k but not cited:           score = 0.5
else:                                                         score = 0.0

# expected_source.sub_category empty (open-ended track) → score = 1.0 (skip)

Threshold 0.5 — retrieved-but-not-cited also passes.

Deterministic · no LLM

Answer Keyword Match

Substring-based keyword recall between the expected and actual answers. Keywords are extracted from expected_output by splitting on whitespace/punctuation, removing Korean stopwords (은/는/이/가/을/를/…) and meta-stopwords (권장, 사라지고, 1학년, 공통교양, …), and dropping tokens shorter than 2 characters. If extraction yields nothing, the item is skipped (score 1.0).

# Verification branch (input matches a positive/negative verification pattern,
# or type == "verification"):
targets = keywords that appear in input (the entity being verified)
score = matched targets in actual / total targets

# General branch:
score = matched keywords in actual / total keywords

_POSITIVE_VERIFY = r"(맞아|맞나요|맞지|포함돼|포함되|있어|들어가|되나요|들어가나요)"
_NEGATIVE_VERIFY = r"(아니지|아니야|없는\s*거|아닌가요)"

Threshold 0.7 — 70% of expected keywords must appear in the actual answer.
